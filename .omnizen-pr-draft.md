## Summary

Initial port of `open-webui/desktop` into Omnizen Desktop. The desktop
app stays a faithful Open WebUI client - multi-connection sidebar,
Spotlight, voice input, Open Terminal, settings all intact - but on
first launch it offers a "Sign in with Omnizen" CTA that routes every
chat completion through `api.omnizen.ai` via a virtual API key the
user never sees. Native binaries for macOS / Windows / Linux ship from
this repo as unsigned alpha; signing comes later.

## Motivation

The hosted browser chat at `chat.omnizen.ai` stays live, but heavy
storage-bound users push noticeable load on our EC2 disk and Postgres.
The desktop port keeps all chat history + uploads + the OpenWebUI
server itself on the user's machine - we only host the gateway at
`api.omnizen.ai`. This is additive: chat.omnizen.ai is not being
retired.

## Design - additive only

- All upstream features still work (multi-connection, Spotlight, voice
  input, Open Terminal, settings).
- "Add Connection" still works - Omnizen is pre-wired as the recommended
  default, not the only option.
- The one subtraction is local inference: `llama.cpp` + the HuggingFace
  model browser are removed because we route to `api.omnizen.ai` and
  don't need an on-device runtime. That removal lives in its own
  commit so an upstream rebase stays clear.

## How sign-in works

1. First launch shows the "Sign in with Omnizen" CTA above the
   existing empty-state.
2. Click → `shell.openExternal(omnizen.ai/connect?code=…)`.
3. Local main-process poll on `/api/cli/auth/poll` resolves once the
   user approves in their browser.
4. Returned `api_key` + `openai_base_url` are encrypted via Electron
   `safeStorage` (macOS keychain / libsecret / DPAPI) and persisted at
   `<userData>/omnizen-creds.bin` (`0o600`).
5. The local OpenWebUI server is restarted with `OPENAI_API_BASE_URLS`
   + `OPENAI_API_KEYS` injected. From that point chat completions land
   at `api.omnizen.ai`.

## Test plan

- [ ] First-launch flow: `npm run dev` from a clean userData dir → CTA
      appears → sign-in opens browser → after approval, local server
      restarts and a chat completion hits `api.omnizen.ai` (verify via
      proxy access log).
- [ ] Sign-out wipes creds; next chat completion no longer routes
      through Omnizen.
- [ ] All upstream features still work: add a second remote connection
      by URL; toggle Open Terminal; trigger Spotlight; trigger voice
      input; switch theme; reset app.
- [ ] Packaged builds: `npm run build:linux` produces
      `omnizen_<ver>_amd64.deb` - `dpkg-deb -I` reports
      `Package: omnizen`, `Maintainer: omnizen.ai`. `npm run
      build:mac` produces `Omnizen-<ver>.dmg` with `mdls -name
      kMDItemCFBundleIdentifier Omnizen.app` returning
      `com.omnizen.desktop`. `npm run build:win` produces a Setup
      `.exe`.
- [ ] Release workflow: push to `release` branch creates a GH release
      under `anchorblock/desktop/releases` with the three-platform
      artifact set (signing skipped, unsigned alpha).
